# ブログ記事レビュースキル

記事の公開前に品質チェックと画像最適化を一括実行するワークフロー。

## 前提

- 対象ファイルは `content/ja/entry/` または `content/en/entry/` 配下の `.md` ファイル。
- ユーザーが対象記事のパスを指定する。指定がなければ確認する。

## ワークフロー

以下のチェックリストをコピーして進捗を管理する:

```
Review Progress:
- [ ] Step 1: フロントマター検証
- [ ] Step 2: 画像最適化チェック
- [ ] Step 3: コンテンツチェック
- [ ] Step 4: レビュー結果の報告
```

---

### Step 1: フロントマター検証

記事ファイルを読み、TOML (`+++`) フロントマターの以下を検証する:

| フィールド | チェック内容 |
|-----------|-------------|
| `categories` | 空でないこと |
| `date` | ISO 8601 + `+09:00` 形式であること |
| `description` | 空でないこと（SEO用の要約） |
| `draft` | `false` であること（公開する場合） |
| `image` | 空でないこと（ヘッダー画像が設定済み） |
| `tags` | 空でないこと |
| `title` | 空でないこと |
| `author` | `"mosuke5"` であること |
| `archive` | 記事年と一致する `["YYYY"]` であること |

問題があれば修正を提案し、ユーザーの承認を得て修正する。

---

### Step 2: 画像最適化チェック

記事内で参照されているすべての画像を対象にチェックする。

#### 2-1: 画像パスの収集

以下から画像パスを抽出する:

- フロントマターの `image` フィールド
- 本文中の `![alt](/image/...)` や `{{< figure src="/image/..." >}}` など

#### 2-2: 存在確認

各パスについて `static/` を付けた実ファイルが存在するか確認する:

```bash
ls -la static/image/<filename>
```

存在しなければ報告する。

#### 2-3: サイズチェックとリサイズ

各画像の幅を確認し、700px超ならリサイズする:

```bash
identify -format "%w" static/image/<filename>
```

700px を超えている場合:

```bash
mogrify -resize 700 static/image/<filename>
```

リサイズ後に再度 `identify` で幅が 700px 以下になったことを確認する。

#### 2-4: 圧縮

ファイル形式に応じて圧縮する:

- **PNG**: `optipng -o7 static/image/<filename>`
- **JPEG**: `jpegtran -copy none -optimize -outfile <file> <file>`

#### 注意

- 700px より小さい画像を拡大しない
- 外部URL（`https://...`）の画像はリサイズ対象外。存在確認のみ行う

---

### Step 3: コンテンツチェック

| チェック項目 | 内容 |
|-------------|------|
| `<!--more-->` | 導入段落の直後に存在すること |
| 外部リンク形式 | `[text](https://...)` 形式の外部リンクがないこと。`external_link` ショートコードか `<a target="_blank">` を使うべき |
| TODO マーカー | `[TODO` が残っていないこと |
| 見出し構造 | `## ` (h2) から始まり、階層が飛んでいないこと |

---

### Step 4: レビュー結果の報告

チェック結果をまとめてユーザーに報告する。以下の形式で整理する:

**修正済み:**
- 自動修正した項目（画像リサイズなど）をリストアップ

**要確認:**
- ユーザーの判断が必要な項目をリストアップ

**問題なし:**
- 正常だった項目を簡潔に報告
